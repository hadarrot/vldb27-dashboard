Engine-specific settings

Neo4j-specific

Neo4j 5.x — full Cypher (GQL-style quantified path patterns) over a client–server driver.

ModeClient–server over the Bolt driver
Cold runsPlan cache cleared before every run (CALL db.clearQueryCaches()) → fully cold
No workaroundsNative QPP (inline EAN constraint) and uncapped *k..k / * — the queries below are the textbook GQL form
Native — query the input graph directly; constraint in WHERE (generate-then-filter)
MATCH p = (s:Account)-[:TRANSFER*k..k]->(t:Account)
WITH [r IN relationships(p) | r.amount] AS amts
WHERE ALL(i IN range(0, size(amts)-2) WHERE amts[i] < amts[i+1])
RETURN count(*)
EAN — each edge is an Event node; constraint checked inline via a quantified hinge motif
MATCH p = (s:EAN_Account)-[:OUT]->(first:EAN_Event)
  ( (n:EAN_Event)-[:IN_]->(acc:EAN_Account)-[:OUT]->(m:EAN_Event)
    WHERE n.amount < m.amount ){k-1,k-1}
RETURN count(p)
Leveled — constraint baked into LIFT edges; query is plain reachability
MATCH (start:Leveled_Stage {level:-1})-[:LIFT*k..k]->(end:Leveled_Stage)
RETURN count(*)

Unbounded: the bounded quantifiers become Kleene *.

Kùzu-specific

Kùzu 0.11.3 — embedded (in-process) engine. Kùzu lacks QPP and caps recursion depth, so EAN and the k>30 queries are expressed differently; counts are still validated identical to Neo4j and to the DP oracle.

ModeEmbedded / in-process (local DB dir, no server); bulk COPY-from-CSV load is timed as the build
Caching caveatNo plan-cache-clear exists — the buffer pool stays warm, so this is warm-buffer latency, not fully cold like Neo4j. No warmups added; the first-run compile cost is the slowest sample the trim drops.
30-hop recursion capVariable-length is capped at 30 hops (open *1.. stops at depth 30 too). k ∈ {32…1024} reached by concatenating ≤30-hop segments — verified exact vs the DP oracle.
TRAIL semanticsRelationship-uniqueness (matches Neo4j; avoids walk-explosion on cycles)
EAN has no QPPHinge is explicitly unrolled per k; Kùzu's planner ceils at ~16 events → EAN works only for k ≤ 8, and unbounded EAN is inexpressible (recorded as a transform T.O.).
Native — generate-then-filter; TRAIL + concatenated ≤30-hop segments for k>30
MATCH p = (s:Account)-[r:TRANSFER* TRAIL k..k]->(t:Account)
WITH properties(rels(p),'amount') AS a            -- spans every segment, in path order
WHERE ALL(i IN range(0, size(a)-2) WHERE a[i+1] < a[i+2])   -- Kùzu lists are 1-indexed
RETURN count(*)
-- k>30: -[*TRAIL 30..30]->(m1)-[*TRAIL 30..30]->(m2)…   unbounded: * TRAIL 1.. (depth ≤ 30)
EAN — explicitly unrolled hinge (no QPP); inline consecutive-event predicates
MATCH (a0:EAN_Account)-[:OUT_]->(e1:EAN_Event)-[:IN_]->(a1:EAN_Account)
      -[:OUT_]->(e2:EAN_Event)-[:IN_]->(a2:EAN_Account)        -- … repeated k times
WHERE e1.amount < e2.amount AND e2.amount < e3.amount AND …    -- inline, per consecutive pair
RETURN count(*)
-- planner ceiling ≈ 16 events ⇒ usable for k ≤ 8; unbounded unsupported
Leveled — reachability on the LIFT DAG; end is reserved in Kùzu → use en
MATCH (st:Leveled_Stage)-[:LIFT*k..k]->(en:Leveled_Stage)
WHERE st.level = -1
RETURN count(*)
-- k>30: concatenate ≤30-hop LIFT segments;   unbounded: *1.. (depth ≤ 30)

As with Neo4j, the Leveled graph is materialized once per constraint and reused for all k; the EAN graph is built once and reused for all (expressible) k; Native uses the input graph directly.
